# Repository Foundations

## Canonical automation

Run local workflow commands from the repo root with PowerShell:

```powershell
.\scripts\dev.ps1 -Task bootstrap-server
.\scripts\dev.ps1 -Task bootstrap-client
.\scripts\dev.ps1 -Task run-server
.\scripts\dev.ps1 -Task check-server
```

The root `Makefile` is only a thin wrapper for users who already have GNU Make installed. The PowerShell script is the source of truth for task automation in this Windows-first repo.

## Shared conventions

- Run commands from the repo root. Do not rely on `cd` into `client/` or `server/` to make config resolve correctly.
- Repository-wide formatting and line-ending defaults live in `.editorconfig`.
- Python tooling config lives in the repo-root `pyproject.toml`.
- Server environment state lives in `server/.env`; relative server paths resolve from `server/`, not from the shell working directory.
- Server tests live under `server/tests/`. Client tests live under `client/test/`.
- Read `docs/CURRENT_MILESTONE.md` before changing setup assumptions or cross-cutting architecture.
- Add shared documentation under `docs/` when you introduce workflow expectations that affect more than one workstream.

## Client scaffold policy

`client/` contains the app source and package manifest. Platform folders are allowed to be generated by `.\scripts\dev.ps1 -Task bootstrap-client` when `android/` or `windows/` are missing. That keeps the repo light while preserving a repeatable setup path for later workers.

## Server scaffold policy

`bootstrap-server` creates `server/.venv` and installs both runtime and dev dependencies. Downstream server work should use that virtual environment so linting, tests, and imports stay consistent across workers.
